# Database Setup (Entity Framework Core + PostgreSQL)

This guide explains how to connect the backend to a PostgreSQL database on
your own machine, and how the EF Core migrations workflow works when
several people are developing against the same schema.

It complements [`environment.md`](./environment.md) (all environment
variables) and [`git-workflow.md`](./git-workflow.md) (branching/PR rules).

---

## 1. Prerequisites

- [.NET 10 SDK](https://dotnet.microsoft.com/download)
- Access to a PostgreSQL database â€” either:
  - **This project's shared database (Supabase)** â€” ask the project lead
    for the database password over a secure/private channel (never
    chat, issues, or commits). The host/port/username are not secret
    and are already filled in for you in step 2 below.
  - **Local PostgreSQL** â€” for working in isolation from the shared
    schema. Install it yourself, or run it with Docker:

    ```bash
    docker run --name moodboard-postgres \
      -e POSTGRES_USER=postgres \
      -e POSTGRES_PASSWORD=postgres \
      -e POSTGRES_DB=moodboard_ai \
      -p 5432:5432 -d postgres:16
    ```

- The `dotnet-ef` CLI tool. This project pins its version in
  `backend/.config/dotnet-tools.json` so everyone uses the same version â€”
  you don't need to (and shouldn't) `dotnet tool install --global
  dotnet-ef` separately. Just restore it (see step 3).

## 2. Configure your own connection string

1. From the **repository root**, copy the example env file if you haven't already:

   ```bash
   cp .env.example .env
   ```

2. Open `.env` and set `CONNECTIONSTRINGS__DEFAULTCONNECTION` to your own
   database:

   ```env
   # Local PostgreSQL (e.g. the Docker command above)
   CONNECTIONSTRINGS__DEFAULTCONNECTION=Host=localhost;Port=5432;Database=moodboard_ai;Username=postgres;Password=postgres

   # Supabase (shared team database) â€” this project's Session Pooler.
   # Get the real password from the project lead over a secure channel
   # (never chat/commits) and put it here â€” never in code or docs.
   CONNECTIONSTRINGS__DEFAULTCONNECTION=Host=aws-0-eu-west-1.pooler.supabase.com;Port=5432;Database=postgres;Username=postgres.fbetndsaukfezfmhjelv;Password=<your-password>;SSL Mode=Require;Trust Server Certificate=true
   ```

   > **Note on Supabase connection modes:** the host/port above
   > (`*.pooler.supabase.com:5432`) is Supabase's **Session Pooler** â€”
   > it behaves like a normal PostgreSQL connection and works with EF
   > Core/Npgsql without extra configuration. Supabase also offers a
   > **Transaction Pooler** on port `6543`, which is faster under high
   > concurrency but doesn't support some features EF Core relies on
   > (e.g. prepared statements) unless you disable them explicitly. Stick
   > to the Session Pooler (port `5432`) shown above unless you have a
   > specific reason to switch.

3. That's it â€” you do **not** need to manually `export` anything or set
   Windows/macOS system environment variables. The backend automatically
   loads `.env` on startup (see `Program.cs` â†’ `LoadDotEnvFile()`): it
   searches the current directory and its parents for a `.env` file, so it
   works no matter whether you run `dotnet run` from the repo root, from
   `backend/`, or from `backend/src/MoodboardAI.Api/`.
4. `.env` is git-ignored â€” never commit it, and never paste real
   credentials into chat, issues, or commits. Each teammate keeps their
   own copy locally.

## 3. Restore the EF Core CLI tool

From the `backend/` folder:

```bash
cd backend
dotnet tool restore
```

This installs the exact `dotnet-ef` version pinned in
`.config/dotnet-tools.json` into the project locally. Prefix every `ef`
command below with `dotnet` (not `dotnet-ef`) so it picks up this local
tool, e.g. `dotnet ef ...`.

---

## 4. Create and apply the initial migration

> Only needs to be done **once** by whoever first sets up the schema. If a
> `backend/src/MoodboardAI.Api/Migrations/` folder already exists and is
> committed to the repo, skip straight to "Applying migrations" below â€”
> you don't need to (and shouldn't) create another initial migration.

From `backend/`, with your `.env` connection string pointing at a reachable database:

```bash
dotnet ef migrations add InitialCreate \
  --project src/MoodboardAI.Api \
  --startup-project src/MoodboardAI.Api
```

This generates the `Migrations/` folder (migration + designer files and a
model snapshot) based on the current `ApplicationDbContext` model
(`Users`, `Interests`, `UserInterests`). **Commit this folder** â€” it must
be shared with the team, not regenerated by each person.

Apply it to your database:

```bash
dotnet ef database update \
  --project src/MoodboardAI.Api \
  --startup-project src/MoodboardAI.Api
```

## 5. Everyday usage (after the first migration exists)

### Applying migrations (most common case)

After pulling changes that include new migrations, just run the app â€”
`Program.cs` calls `dbContext.Database.Migrate()` automatically on
startup **in the Development environment**, so your local schema is kept
up to date without running any EF command by hand:

```bash
dotnet run --project src/MoodboardAI.Api
```

If you'd rather apply migrations without starting the API, you can still
run the explicit command from step 4:

```bash
dotnet ef database update --project src/MoodboardAI.Api --startup-project src/MoodboardAI.Api
```

### Adding a new migration (when you change an entity/DbContext)

Whenever you add/change an entity, a property, or the `OnModelCreating`
configuration in `ApplicationDbContext`, generate a new migration
describing the change:

```bash
dotnet ef migrations add DescribeYourChangeHere \
  --project src/MoodboardAI.Api \
  --startup-project src/MoodboardAI.Api
```

Commit the newly generated files in `Migrations/` together with your
entity changes in the same PR, then run `dotnet run` (or `dotnet ef
database update`) to apply it locally before testing.

### Verifying the connection works

Once the app is running, open Swagger (`http://localhost:5246/swagger` by
default) and call `GET /api/interests`. A `200 OK` with the 10 seeded
interests confirms EF Core successfully connected to and read from your
database.

## 6. Troubleshooting

| Symptom | Likely cause / fix |
| --- | --- |
| App throws a Npgsql connection exception on startup | Database isn't running/reachable, or the connection string in `.env` is wrong. Double-check host/port/username/password. |
| `.env` values don't seem to apply | Make sure the file is named exactly `.env` (not `.env.txt`) and sits somewhere at or above the directory you're running `dotnet` from. Restart the app after editing it. |
| `dotnet ef` says "No executable found matching command 'dotnet-ef'" | Run `dotnet tool restore` from `backend/` first. |
| Supabase connection fails with an SSL error | Add `SSL Mode=Require;Trust Server Certificate=true` to your connection string. |
| `dotnet ef migrations add` produces unexpected changes to old migrations | Someone else already changed the model since your last pull â€” run `git pull`, restore packages, and try again instead of committing a diverging snapshot. |
| Migration fails because a table/type already exists | The database already has this schema from a previous run. Either drop the affected tables or, for a local-only DB, drop and recreate the database. |
